# Система контроля качества для e-commerce на основе кросс-доменной адаптации методов обнаружения фейков с использованием больших мультимодальных моделей

Дипломный проект по разработке системы детекции контрафактных товаров на маркетплейсе на основе иерархической кросс-доменной модели **HCDM** (Hierarchical Cross-Domain Model), агрегирующей методы из четырёх предметных областей: социальных сетей, мобильных приложений, недвижимости и финтеха.

## Ключевые результаты

Итоговая модель **HCDM** на едином seller-based-сплите датасета Ozon eCup 2025 (n_test = 58 410):

| Метрика | Baseline CatBoost | CDSM v3 (предш.) | **HCDM (финал)** |
| --- | --- | --- | --- |
| ROC-AUC | 0,9051 | 0,9450 | **0,9492** |
| PR-AUC | 0,6231 | 0,7604 | **0,8044** |
| Recall @ Precision ≥ 0,9 | 0,0145 | 0,0828 | **0,2068** |

Превосходство HCDM над CDSM v3 подтверждено многосидовой Wolpert-валидацией: 5 из 5 random seeds, p < 0,05 по парному бутстрап-тесту.

## Архитектура HCDM

HCDM — выпуклая композиция выходов четырёх независимо обученных доменных моделей с фиксированными неотрицательными весами:

```
p_HCDM = 0,875·p_social + 0,075·p_mobile + 0,025·p_realestate + 0,025·p_fintech
```

| Канал | Модель | Вес | PR-AUC | Ключевой методический вклад |
| --- | --- | --- | --- | --- |
| Социальные сети | MMD-Mode-3 | 0,875 | 0,8052 | Многорежимный ансамбль (3 режима глубины) + мета-модель Wolpert |
| Мобильные приложения | Mobile-Adapted | 0,075 | 0,6453 | Эмбеддинг-расстояния (CLIP + SVD/TF-IDF) |
| Недвижимость | RealEstate-Adapted | 0,025 | 0,6724 | `price_too_high` (FADAML), typosquat-паттерны |
| Финтех | Fintech-Adapted | 0,025 | 0,6817 | Калибровка дисбаланса (SMOTE, undersampling) |

Веса подобраны grid search'ом по симплексу с шагом 0,025 на отдельной валидационной выборке (оптимизация среднего ранга по PR-AUC и R@P ≥ 0,9). Ablation study подтверждает необходимость каждого из четырёх компонентов.

## Программная система (прототип)

Реализована production-готовая двухканальная архитектура: канал ранжирования (непрерывный `p_HCDM` для очереди модерации) + канал автоматических действий (бинаризация по порогу `τ_auto` для precision ≥ 0,9).

### Технологический стек

| Компонент | Технология | Ответственный |
| --- | --- | --- |
| REST API | FastAPI (ASGI, Pydantic, OpenAPI) | Азимова |
| Очередь задач | RabbitMQ (AMQP, dead-letter exchange) | Мкоян |
| Хранилище | PostgreSQL 15 + SQLAlchemy 2.x (asyncpg) + Alembic | Бахтиарова |
| Веб-интерфейс | HTML + JavaScript (без фреймворков), деплой на Timeweb | Красовская |
| Оркестрация | Docker Compose | 


### Эндпойнты API

- `POST /predict` — приём карточки товара, асинхронный инференс
- `GET /result/{task_id}` — получение результата по идентификатору задачи
- `POST /feedback/{prediction_id}` — обратная связь модератора

### Подсистема хранения

Таблицы: `predictions` (с покомпонентной декомпозицией `p_social`, `p_mobile`, `p_realestate`, `p_fintech`), `feedback`, `seller_profiles` (LOO-кодирование), `model_versions`, `processing_log`. 

### Характеристики прототипа

| Параметр | Целевое значение | Факт на dev-стенде |
| --- | --- | --- |
| Latency P95 (1 карточка) | ≤ 500 мс | ~250 мс |
| Throughput | ≥ 100 RPS | ~280 RPS (4 worker'а) |
| Доступность | 99,5% | — |
| Error rate (sustained load) | — | 0% |

## Научные выводы

1. **О хрупкости R@P.** Метрика Recall@Precision ≥ 0,9 при сильном дисбалансе классов хрупка (σ ≈ 0,03 по 5 сидам); PR-AUC устойчивее и предпочтительнее для сравнения моделей.
2. **О переносимости из социального домена.** Структурное сходство мультимодальной задачи определяет переносимость сильнее, чем предметная близость источника к e-commerce.
3. **Об ансамблевой адаптации.** Многорежимная архитектура MMD-Thinker эффективно переносима из нейросетевой парадигмы в ансамбль табличных моделей.
4. **О diversifier-эффекте.** Convex-композиция независимых доменных моделей даёт значимый прирост за счёт ортогональности SHAP-векторов (средний |r| = 0,12 между каналами).
5. **Об ограничениях нейросетей.** На имеющемся объёме данных усложнение нейроархитектур (MoE, Co-Attention) ухудшает качество — согласуется с наблюдениями Grinsztajn et al. о преимуществах tree-based моделей на табличных данных.
6. **О seller-based разбиении.** Случайный сплит приводит к утечке через `seller_id` и завышению R@P ≥ 0,9 на порядок.
7. **О роли SHAP.** SHAP-разложение даёт не только интерпретируемость, но и количественную оценку ортогональности каналов композиционной модели.
8. **О двухканальной production-архитектуре.** Раздельные режимы ранжирования и автоматических действий предпочтительнее одноканального решения при асимметричных требованиях.
